Adoption Analysis: go-error-family in go-filewatcher

Research Date: 2026-06-01 · Module: github.com/larsartmann/go-filewatcher/v2 · Candidate: github.com/larsartmann/go-error-family

1. Executive Summary

This report evaluates whether go-filewatcher should adopt go-error-family as its structured error system. The current system is home-grown (WatcherError, ErrorCategory, 10 sentinel errors). The candidate is a standalone library with five behavioral families, exit codes, diagnostics, and a template-driven CLI boundary.

Verdict: Do not adopt at this time. The migration cost exceeds the value delivered for a library whose errors are almost entirely internal. A lighter integration path (diagnostics only, or a future v3) is preferred.

3. PRO Advantages

  • Richer semantic taxonomy. Five families vs. three categories gives callers finer-grained recovery logic. A Conflict (version mismatch) deserves different handling than a Rejection (bad input).
  • Built-in exit codes. go-filewatcher is a library, but its consumers are often CLI tools. Returning a family-derived exit code (e.g. 75 for Transient) is valuable for shell scripts and CI pipelines.
  • Template-driven user messages. The MessageTemplate system (What/Why/Fix/WayOut) produces production-ready CLI output without each consumer reinventing formatting.
  • Diagnostic rules. The diagnose package can automatically detect common root causes (missing files, unreachable hosts, Git issues, Postgres down). This is especially useful for a file watcher where “path not found” often means a symlink is broken or a network mount dropped.
  • Third-party error registration. RegisterClassification lets consumers teach the system about fsnotify or OS errors without forking the library.
  • Composition primitive. Compose() provides a standardized way to surface multiple watch-path failures without custom aggregation logic.
  • Ecosystem alignment. Adopting a shared error vocabulary across LarsArtmann projects reduces cognitive load for engineers working across repos.
  • Better nil handling. Classify(nil) returns Rejection (caller’s fault) rather than Unknown, making nil-check bugs explicit.

4. CONTRA Blockers & Costs

  • Massive breaking change. WatcherError, ErrorCategory, ErrorCode, all 10 sentinel errors, ErrorContext, and ErrorHandler are declared Stable in API_STABILITY.md. A major version bump (v3) would be required.
  • Loss of typed fields. go-filewatcher’s WatcherError has first-class Op (OpString) and Path fields. go-error-family flattens everything into map[string]string. Callers lose compile-time field access and must use string keys.
  • Loss of stack traces. go-error-family has no built-in stack capture. go-filewatcher currently auto-captures debug.Stack() on every NewWatcherError. This would need to be reimplemented as a wrapper or extension.
  • Handler pattern mismatch. go-filewatcher’s async error pipeline (channel + ErrorHandler callback with ErrorContext) does not map to go-error-family’s synchronous HandleError(err) CLI boundary. Rewiring the Watcher.handleError dispatcher is non-trivial.
  • Context type restriction. go-error-family enforces map[string]string values only. go-filewatcher’s ErrorContext carries an *Event pointer and a bool. These would need serialization or a parallel side-channel.
  • Naming convention churn. Error codes migrate from SNAKE_CASE (PATH_NOT_FOUND) to dot-notation (file.not_found). Every consumer doing programmatic matching breaks.
  • False precision in families. Most go-filewatcher errors are simple: path missing, watcher closed, fsnotify failed. The five-family model is overkill for a domain where “permanent vs. transient” is usually sufficient.
  • Diagnostics are heavy for a library. The diagnose subsystem shells out to commands (pg_isready, git status, DNS lookups). For a library that runs inside long-lived daemons, unexpected subprocess spawning is a liability.
  • New dependency. Despite being same-org, adding a module dependency increases supply-chain surface and vendor-hash maintenance in flake.nix.
  • Test rewrite burden. errors_test.go (500+ lines), watcher_test.go, watcher_coverage_test.go, and helper files all reference WatcherError and ErrorCategory directly.

5. Neutral Considerations Context

  • Both libraries target Go 1.26+ and use errors.AsType[T]().
  • Both are zero-dependency (stdlib only) — dependency weight is not a differentiator.
  • Both support errors.Is / errors.As chaining.
  • go-error-family’s Classify fail-open default (Transient) matches go-filewatcher’s current CategoryUnknown behavior in spirit.
  • go-filewatcher’s pre-existing TODOs suggest making stack capture opt-in; this reduces the “loss of stacks” concern because the current behavior is already flagged as expensive and undesirable.

Final Verdict

Do not adopt go-error-family in go-filewatcher v2.

The benefits (richer taxonomy, diagnostics, exit codes) are real but are primarily valued by CLI applications, not by a file-watching library whose errors are consumed programmatically via channels and callbacks. The adoption would force a v3 major version, rewrite hundreds of lines of stable tests, and discard typed fields (Op, Path) that are domain-specific to file watching.

Recommended path:

  1. Maintain the current error system for v2.
  2. If a v3 is ever planned, design it around go-error-family primitives from day one.
  3. Optionally, import go-error-family/diagnose as an optional diagnostic helper in handleError (behind a build tag or option) without replacing the core error types.
