go-filewatcher Improvement Plan

Addressing the inotify scalability wall & production readiness for v0.4.0

Date: 2026-06-03 Current: v0.3.0 Target: v0.4.0 Status: Proposal

Executive Summary

The projects-management-automation service watches /home/lars/projects (113k+ directories, 278 git repos) through go-filewatcher. Despite ignore patterns filtering .git, node_modules, vendor etc., ~76k directories survive the filter and require individual inotify watches. This causes ENOSPC ("no space left on device") — a misleading kernel error meaning the max_user_watches limit is exhausted.

This plan addresses the root causes across three dimensions: inotify budget reduction (most urgent), architectural improvements to the walk engine, and API enhancements for consumer ergonomics.

Root Cause Analysis

Why 76k Watches?

Directory Subdirs (post-filter) Notes
forks/ 27,480 Cloned repos, likely not actively edited
kotlin/ 22,001 Kotlin projects with deep source trees
faceswap/ 6,116 ML project with large Python tree
chats/ 4,706 Chat application projects
Other 270+ repos ~16k Long tail of small-to-medium projects
Problem 1: No .gitignore awareness. The library has hardcoded DefaultIgnoreDirs and user-configured WithIgnoreDirs, but never reads .gitignore files. Build artifacts, generated code, IDE configs, and project-specific exclusions are all watched needlessly.
Problem 2: Eager, synchronous, all-at-once watching. Watch() calls addPath() for every path, which calls walkAndAddPaths(), which calls filepath.WalkDir() synchronously — adding every directory to inotify before returning. No batching, no lazy loading, no backpressure.
Problem 3: shouldSkipDir only matches directory names, not paths. You cannot exclude forks/ or archived/ at the top level without also excluding any directory named "forks" or "archived" anywhere in the tree.
Problem 4: No graceful degradation. When fswatcher.Add() fails with ENOSPC, walkDirFunc returns the error, which stops the entire filepath.WalkDir traversal — the watcher fails to start at all. It should log a warning and continue.

Architecture: Current vs Proposed

Current Pipeline

Watch() └─ for each path: addPath(root) └─ walkAndAddPaths(root) ← synchronous, blocking └─ filepath.WalkDir() └─ walkDirFunc() ├─ shouldSkipDir(name) ← name-only, no .gitignore └─ fswatcher.Add(path) ← one inotify watch per dir ← ENOSPC kills entire walk

Proposed Pipeline

Watch() └─ for each path: addPath(root) ├─ loadGitignore(root) ← read .gitignore at repo root └─ walkAndAddPaths(root) └─ filepath.WalkDir() └─ walkDirFunc() ├─ shouldSkipDir(name) ← enriched with gitignore match ├─ skipPath(path) ← path-level exclusions └─ fswatcher.Add(path) ← continues on ENOSPC // Lazy loading (optional) handleNewDirectory() └─ only walks subtree on first event ← reduces initial burst

Work Items

Critical W1: Add .gitignore-aware walk filtering

Problem: 76k directories watched when ~8k would suffice. Each .gitignore excludes build dirs, generated code, and project-specific artifacts that DefaultIgnoreDirs doesn't cover.

Solution: During walkDirFunc, detect .gitignore files (by checking if a .git parent exists) and apply their rules to skip directories. Use composition with a gitignore library as documented in the gogenfilter gitignore guide.

Implementation

  • Add a gitignoreCache to Watcher: map[string]*gitignore.GitIgnore keyed by repo root
  • During walkDirFunc: if .gitignore file exists in current dir, load and cache it
  • Check directory path against all applicable gitignore rules before calling fswatcher.Add()
  • New option: WithGitignore(enabled bool) (default: true)

Dependency

Add github.com/go-git/go-git/v6/plumbing/format/gitignore (full git spec compliance) — or keep it lighter with github.com/sabhiram/go-gitignore (zero transitive deps). The gogenfilter docs already evaluate both.

Estimated impact

~50-70% reduction in inotify watches. Most Go projects have bin/, tmp/, coverage/, *.egg-info/, .tox/, .venv/ etc. in their .gitignore that aren't in DefaultIgnoreDirs.

Files Changed

watcher_walk.go Load gitignore in walkDirFunc, check before Add()
options.go Add WithGitignore(bool)
watcher.go Add gitignoreCache field to Watcher
filter.go Add FilterGitignore() for event-level filtering
go.mod Add gitignore library dependency

Critical W2: Graceful ENOSPC handling — continue on watch failure

Problem: In walkDirFunc (watcher_walk.go:90-93), if fswatcher.Add() returns ENOSPC, the error propagates up through filepath.WalkDir and Watch() returns an error — the watcher fails to start entirely.

Solution: On ENOSPC (or any fswatcher.Add() error), log a warning and continue walking. The watcher starts in a degraded state — watching what it can — rather than failing entirely.

Implementation

// In walkDirFunc, replace:
addErr := w.fswatcher.Add(path)
if addErr != nil {
    return fmt.Errorf("watching path %q: %w", path, addErr)
}

// With:
addErr := w.fswatcher.Add(path)
if addErr != nil {
    w.handleError(ErrorContext{
        Operation: "add-path",
        Path: path,
        Retryable: true,
    }, fmt.Errorf("watching path %q: %w", path, addErr))
    // Don't return the error — continue walking other dirs
    return nil
}

Also add a watchErrors atomic counter to Stats so consumers can detect degraded mode and alert.

Zero breaking changes. Watcher becomes resilient to inotify limits instead of failing hard. PMA can continue watching active repos even when forks/ exhausts the budget.

High W3: Path-level exclusion patterns (WithExcludePaths)

Problem: shouldSkipDir matches directory names only. You cannot exclude /home/lars/projects/forks without also excluding any directory named "forks" anywhere. FilterExcludePaths exists for events but not for walk-time filtering.

Solution: Add walk-time path-level exclusion. A new option WithExcludePaths(paths ...string) that stores absolute paths and skips them during walkDirFunc — before calling fswatcher.Add().

Implementation

  • Add excludePaths map[string]struct{} to Watcher (populated from option, normalized to absolute paths)
  • In walkDirFunc, check if the current path is a prefix of any excluded path (or exact match)
  • Use filepath.Rel() or prefix matching for efficient checks
// options.go
func WithExcludePaths(paths ...string) Option {
    return func(w *Watcher) {
        for _, p := range paths {
            abs, _ := filepath.Abs(p)
            w.excludePaths[abs] = struct{}{}
        }
    }
}

// watcher_walk.go walkDirFunc
if w.shouldExcludePath(path) {
    return filepath.SkipDir
}
Immediate fix for PMA: Add WithExcludePaths("/home/lars/projects/forks", "/home/lars/projects/archived") and cut ~30k watches.

High W4: Batched / lazy watch registration

Problem: Watch() blocks until every directory is added to inotify. For 76k directories, this means a multi-second startup where Watch() doesn't return. No events can be processed during this time.

Solution: Two complementary approaches:

Option A: Batched registration (simpler)

Walk the tree, collect directories into batches, add watches in chunks with runtime.Gosched() between batches. Watch() returns immediately, a background goroutine adds watches progressively.

Option B: Lazy loading (more efficient)

Only watch top-level directories initially. When a Create event fires in a subdirectory, add watches for its children on demand. Requires fsnotify to detect new dirs in unwatched subtrees — may miss events in deeply nested directories created between scans.

Recommendation

Approach
Option A (batched) for v0.4.0. Option B (lazy) as future work.
Rationale
Batched is simpler, doesn't miss events, and solves the startup latency. Lazy loading has edge cases with deeply nested new directories. 
// watcher_walk.go
const watchBatchSize = 1000

func (w *Watcher) walkAndAddPaths(root RootPath) error {
    var batch []string

    err := filepath.WalkDir(root.Get(), func(path string, d os.DirEntry, walkErr error) error {
        // ... existing skip logic ...

        batch = append(batch, path)

        if len(batch) >= watchBatchSize {
            w.addBatch(batch)
            batch = batch[:0]
            runtime.Gosched() // yield to event processing
        }

        return nil
    })

    // Flush remaining
    if len(batch) > 0 {
        w.addBatch(batch)
    }

    return err
}

func (w *Watcher) addBatch(paths []string) {
    for _, p := range paths {
        if err := w.fswatcher.Add(p); err != nil {
            w.handleError(ErrorContext{Operation: "add-path", Path: p, Retryable: true}, err)
        }
    }
}

Medium W5: Inotify budget awareness

Problem: The library has no awareness of system inotify limits. It blindly adds watches until the kernel returns ENOSPC.

Solution: Read /proc/sys/fs/inotify/max_user_watches at startup and track watches against the budget. When approaching the limit (e.g., 90%), stop adding watches and log a warning with actionable advice.

Implementation

  • New Stats field: WatchLimit int (populated from /proc/sys/fs/inotify/max_user_watches on Linux)
  • New Stats field: WatchBudgetUsed float64 (percentage)
  • Option: WithMaxWatches(n int) to override the system limit (for testing or conservative usage)
  • In walkDirFunc: check len(w.watchList) >= w.maxWatches before adding 
// options.go
func WithMaxWatches(n int) Option {
    return func(w *Watcher) {
        w.maxWatches = n
    }
}

// watcher_walk.go — in walkDirFunc
if w.maxWatches > 0 && len(w.watchList) >= w.maxWatches {
    return filepath.SkipDir // budget exhausted
}
Deterministic behavior. Instead of random ENOSPC failures, the library respects a budget and degrades gracefully. Consumers get early warnings via stats.

Medium W6: Remove() doesn't remove subdirectory watches

Problem: Watcher.Remove(path) (watcher.go:453-482) calls w.fswatcher.Remove(abs) for the top-level path only. When Add() added 500 subdirectories recursively, Remove() only removes the root — the 500 subdirectory watches leak until Close().

Solution: When removing a path that was added recursively, also remove all subdirectory watches under that path.

Implementation

func (w *Watcher) Remove(path string) error {
    // ... existing validation ...

    abs, _ := filepath.Abs(path)

    // Remove the path itself
    _ = w.fswatcher.Remove(abs)

    // Remove all subdirectory watches under this path
    prefix := abs + string(filepath.Separator)
    var remaining []string
    for _, p := range w.watchList {
        if p == abs || strings.HasPrefix(p, prefix) {
            _ = w.fswatcher.Remove(p)
        } else {
            remaining = append(remaining, p)
        }
    }
    w.watchList = remaining

    return nil
}

Low W7: Watch restart / reset without re-creating the watcher

Problem: Once a watcher is closed, it cannot be restarted — you must call New() + Watch() again. The TODO list already notes "Can be done via Close + New + Watch" but that loses all configuration.

Solution: Add Reset() method that clears state but preserves configuration, allowing Watch() to be called again after a Close().

  • Reset state flags (clear flagClosed)
  • Re-create the internal fsnotify.Watcher
  • Clear watchList, reset counters
  • Preserve: paths, filters, middleware, debounce, all options
Consumer ergonomics. PMA can reset the watcher on config reload without rebuilding from scratch.

Breaking Changes

None. All changes are additive. Default behavior is preserved — WithGitignore(true) is the only default that changes, but it's a strict improvement (fewer false positives, no false negatives).

Design Decisions

Decision Chosen Alternatives Considered
Gitignore library Start with sabhiram/go-gitignore (zero deps) go-git/plumbing/format/gitignore (alpha, heavy transitive deps)
Gitignore loading timing Load during walk (walkDirFunc) Pre-scan all .gitignore files before walk (slower startup, more accurate)
ENOSPC behavior Log warning + continue walking Return error (current), fail-open (ignore completely), panic
Watch batching strategy Background goroutine with batches Lazy on-demand watching (edge cases with nested dirs)
Inotify limit source Read from /proc/sys/fs/inotify/max_user_watches on Linux Require user to pass it, auto-detect via syscall
Path exclusion semantics Prefix match on absolute paths Glob patterns (more flexible but slower), exact match only

Open Questions

  1. Gitignore default: Should WithGitignore(true) be the default? It changes behavior for existing consumers. Recommendation: yes, with release note.
  2. Windows support: The inotify budget (W5) is Linux-specific. Should we use ReadDirectoryChangesW limits on Windows? Recommendation: Linux-only for v0.4.0, add Windows/macOS in v0.5.0.
  3. Gitignore performance: Loading + matching gitignore rules per directory has cost. Should we benchmark the overhead? Recommendation: yes, add a benchmark with 1000+ directories.
  4. Batched registration ordering: Should higher-priority paths (closer to root) be watched first? Recommendation: yes, walk order is already depth-first which naturally prioritizes shallow paths.
