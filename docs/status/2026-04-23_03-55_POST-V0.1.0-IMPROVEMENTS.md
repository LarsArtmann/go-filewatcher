**What:** Removed `UsesPerPathKeys()` and `Close()` from `DebouncerInterface`.

**Why:** `UsesPerPathKeys()` leaked implementation details through the interface â€” the caller had to ask the debouncer what kind it was. `Close()` was a redundant alias for `Stop()` that confused users. Both are code smells identified in the previous status report.

**How (Option B):** Always pass `DebounceKey(path)` to `Debounce()`. `GlobalDebouncer` already ignores the key. No behavioral change.

## d) TOTALLY FUCKED UP! đź”Ą

### watcher_walk_test.go Compile Errors (FIXED in-session)

**What happened:** Initial `watcher_walk_test.go` used `os.Stat()` which returns `fs.FileInfo`, but `walkDirFunc` expects `os.DirEntry` (a different interface). The two types are not interchangeable.

**How fixed:** Replaced `os.Stat()` calls with `os.ReadDir()` which returns `[]os.DirEntry`, then searched for the matching entry by name.

**Lesson:** Always check Go's `filepath.WalkDir` callback signature â€” it uses `fs.DirEntry`, not `fs.FileInfo`.

### 3. Interface Design Pattern

**Lesson from the DebouncerInterface cleanup:** When an interface method exists solely to branch behavior in the caller, it's a code smell. The key insight was that `GlobalDebouncer` already ignores the key â€” so we should always pass it and let the implementation decide.

**Pattern to follow:**
```go
// GOOD: Caller passes all context, implementation decides what to use
Debounce(key DebounceKey, fn func())

// BAD: Caller queries implementation details
if debouncer.UsesPerPathKeys() { ... }
```

### 4. Test Flakiness Prevention

**Lesson:** Debounce tests must use a debounce window significantly larger than the total write duration. The old test had 100ms debounce with 5Ă—10ms = 50ms total writes â€” too close for comfort on slow CI.

**Rule of thumb:** `debounce_window â‰Ą 4 Ă— total_write_duration`
